# Autonomous CMO reasoning loop and portfolio scheduler

The change adds an Executive Brain, specialist roster, durable website memory, portfolio discovery, recurring CMO cycles, mission execution, reporting, and an Executive UI. It forms a recognizable sense → analyze → prioritize → plan → govern → execute → verify → learn loop, and recommendation-only missions do not directly call publishing. The shipping path is unsafe because site identity is lost at the Digital Twin/publishing boundary, manual APIs bypass scheduler policy, and process-global state lacks tenant ownership. Strategic outputs also present generated sample signals and formulaic forecasts as observed business evidence.

**Watch for:** **confirmed** cross-site fix selection and single-site publishing credentials can direct an autonomous cycle at the wrong website; **confirmed** unauthenticated/manual entry points can override website AI, automation, memory, and governance policy; **confirmed** `Website.agent_config` updates are lossy, non-atomic, and only partially bounded; **confirmed** scheduler restart and deployment concurrency can duplicate cycles; **confirmed** synthetic observations, completion counts, and forecasts overstate what was measured or verified.

## High-level view

**Confirmed:** the scheduler carries `website_id`, but execution falls back to the application’s single `Subsystems` bundle. A cycle for one site enumerates every issue and fix in the tenant and publishes through one globally configured WordPress adapter, so the site boundary does not survive execution.

**Confirmed:** scheduler discovery checks `ai_enabled`, `memory_enabled`, `automation_enabled`, and `approval_mode`, but manual run, report, scheduler-control, safe-apply, and mission-execution endpoints do not share that gate. Request bodies can select autonomous mode, and the new mutating/control endpoints have no authentication or authorization dependency.

**Confirmed:** CMO memory is scoped when repository reads use `(tenant_id, website_id)`, but the whole `agent_config` JSON document is rewritten without optimistic locking and the public onboarding schema cannot preserve the new CMO keys. Several profile collections and duplicated roadmap payloads remain unbounded, while process-local fallbacks, runs, outcomes, goals, and assessments have no retention policy.

**Confirmed:** the cadence driver is a process-local singleton started during app construction. It has no distributed lease or per-site in-flight guard, and its stop/start handling can leave an old worker alive when a replacement starts; transient discovery errors are converted into an empty portfolio and remove discovered registrations.

**Confirmed:** backlink, competitor, GEO, and local SEO inputs are fixed or seeded sample data, while ranking, traffic, lead, revenue, budget, and completion forecasts are formulaic. The UI renders these as website observations and expected results, and reporting treats accepted recommendations without handlers as verified completed actions.

**Confirmed:** the provider-neutral gateway is described in metadata but is not invoked by CMO planning or specialist reasoning. The implementation also creates a second agentic/API stack in the root app while the nested web workspace uses a different composition root that already owns agentic and analytics routes.

## Website identity disappears before fixes reach WordPress

**Confirmed.** `CMOCoordinator.connected_sites` in `packages/api/api/cmo_cycle.py` retains `website_id`, but `_crawl_actions` and `_approve_one` in `packages/api/api/agent_router.py` switch to `request.app.state.subsystems`, use `subs.tenant_id`, call tenant-wide `list_active_issues`/`list_fixes`, and approve matching fixes without filtering to the website being cycled.

`_resolve_subsystems` and `_mount_onboarding` in `packages/api/api/app.py` construct a single configured `WordPressClient`; `AgentContext.subsystems`, `_tool_crawl_site`, and `_tool_apply_fix` in `packages/api/api/agent_tools.py` also use that global bundle instead of the selected onboarding connection and encrypted credentials:

```text
website A discovered with website_id A
        |
        v
crawl target A -> tenant-wide issues/fixes for A + B
        |
        v
global governance + globally configured WordPress client
```

An autonomous cycle can select a pending low-risk fix belonging to website B while running website A, then send it through the WordPress adapter configured in process settings. The execution context needs a required tenant/site identity, site-filtered Digital Twin reads, and a publishing/governance handler resolved from that website’s verified connection.

## Manual CMO controls bypass site policy and authorization

**Confirmed.** `CMOCoordinator.connected_sites` makes AI and memory flags eligibility gates and downgrades autonomous mode when `automation_enabled=False`. `_run_cycle` instead gives body `mode` precedence, so `website_id` plus `mode="autonomous"` bypasses that downgrade. `executive_plan` and `executive_report_endpoint` can analyze and persist without checking AI or memory flags; the GET report endpoint can write memory and audit. `execute_mission` checks only `approval_mode`, not the automation brake, and immediately calls `governance.approve_fix` for a `fix_id` even in Approval mode.

The scheduler start/stop/register/tick endpoints, run creation, bulk safe apply, action approval, goals update, and mission execute endpoints in `build_agent_router` have no authentication or authorization dependency. Actors and rationales are client-controlled, scheduler state is global, and arbitrary targets can be registered for recurring crawl work. `Executive.tsx::MissionRow` labels the operation “Route mission,” while the backend may publish immediately. A fail-closed site policy/RBAC check must cover every manual and scheduled entry point, and routing must remain distinct from human approval.

**Confirmed.** `onboarding.schemas.ApprovalMode` exposes `human`, `auto`, and `scheduled`, while CMO uses `advisory`, `approval`, and `autonomous`. No onboarding value selects Advisory, and `scheduled` silently normalizes to Approval. Use one canonical persisted enum with an explicit legacy migration.

## Process-global registries leak ownership and grow without retention

**Confirmed.** `_RUNS` stores tenant IDs, but list/get/approve/reject/apply endpoints do not filter or verify ownership. `AgentScheduler.status` exposes all registrations; `_find_key` matches only domain or website ID, and unregister/tick do not provide tenant context. A caller who knows another identifier can read or mutate it across process tenants.

`executive_brain.py` scopes goals and assessments by tenant plus normalized domain, not `website_id`; distinct website records sharing a host or differing only by path collide because `_resolve_domain` discards paths. `_FALLBACK`, `_RUNS`, `OUTCOMES`, `_GOALS`, and `_ASSESSMENTS` are unbounded globals. `OUTCOMES` is presented as future-prioritization learning but `score_mission` never consumes it, and the data disappears on restart.

## `Website.agent_config` is not a safe memory store yet

**Confirmed.** `CMOMemoryStore.save` reads a detached Website and later calls `OnboardingRepository.update_website` in a separate transaction. Concurrent assessments, profile updates, mission outcomes, or settings edits can overwrite each other and unrelated configuration. The website update commits before `record_audit` runs separately; if audit insertion fails, the database mutation remains durable and unaudited while the returned copy is labeled ephemeral.

**Confirmed.** `WebsiteUpdate.agent_config` is the fixed `AgentConfig` schema, which has neither `executive_cmo` nor `executive_cmo_schedule`. `routes.update_website` replaces JSON with `model_dump()`, so a normal settings save can erase CMO memory and the schedule read by `connected_sites` cannot be authored through the public API.

**Confirmed.** Performance, cycle, strategy, and `content_history` lists are bounded, but products, competitors, seasonal opportunities, published pages/blogs, internal links, conversion funnels, `strategy_stats`, and individual values are not. `current_roadmap` persists full missions twice through `today`/`daily`, `next_7_days`/`weekly`, and the other aliases. Bound collection and value sizes, persist one roadmap representation, and make memory plus audit one version-checked transaction.

## Cadence lifecycle can produce duplicate workers and destructive discovery

**Confirmed.** `configure_cmo_scheduler` starts a thread during `create_app` and only registers shutdown. Each server process creates an independent scheduler without a distributed lease or idempotency key. `AgentScheduler.stop` waits two seconds and clears `_thread` even if a cycle still runs; `start` then clears the same stop event, allowing the retired worker and its replacement to process concurrently. `tick_now` can overlap the worker because no per-site in-flight marker exists.

**Confirmed.** `CMOCoordinator.connected_sites` converts repository failures to `[]`; `sync_discovered` interprets that as an empty portfolio and removes all discovered registrations. One transient database error clears the schedule until a later successful discovery. A slow synchronous crawl also blocks every other site in the sole worker. Scheduler ownership belongs in application lifespan, with failure-preserving discovery, per-site idempotency/locks, a datastore lease, and bounded dispatch.

## Generated samples are presented as sensed business conditions

**Confirmed.** `_tool_backlink_audit` builds a fixed referring-domain list, `_tool_competitor_gap` defaults every site to Wix/Squarespace/Shopify, and `_tool_brand_visibility` derives AI citations from a domain hash. `engines.shared.local_seo_data` supplies the local knowledge object used by technical and content specialists. These are demo fixtures, while `Executive.tsx` describes continuous website inspection and labels changes as conditions detected since the prior cycle.

`executive_brain.py` produces fixed ranking bands, traffic percentages, leads from a score divisor, budget bands, completion estimates, and revenue ranges without rank/traffic baselines, conversion rates, attribution windows, or delivery capacity. `executive_report.py` defines goal progress as `marketing_score / 85` and labels it on track/improving for traffic, leads, or sales. Until real signals are connected, outputs need `source: simulated`, confidence/provenance, and “not measured” states; goal progress requires actual metrics and time-bound targets.

## Recommendation-only missions do not write, but reporting says they completed

Recommendation-only branches in `execute_mission` and `_approve_one` do not call publishing. **Confirmed concern:** autonomous runs, the UI KPI, `AgentScheduler._run_one`, and `executive_report.build_report` aggregate `accepted` with `applied`, and the narrative calls the total “verified completed action(s).” A recommendation with no handler is reported as completed even though no execution occurred.

**Confirmed.** Verification immediately reruns deterministic analysis and uses a generic health-score delta for every mission. Zero delta records no result, while an immediate negative delta becomes a failed strategy; delayed ranking, traffic, lead, and revenue effects cannot be learned this way. The loop outcome store is not consumed by prioritization. Recorded mission memory is attached to the mission while the assessment’s `memory` stays stale, so an invalidated report can show the old verified-outcome count until another plan rebuilds the assessment.

## Gateway metadata is not gateway-routed intelligence

**Confirmed.** `cmo_memory.provider_descriptor` reads `app.state.intelligence.provider` and emits a gateway label, but no Executive Brain or specialist path invokes that provider, prompt registry, telemetry, cost controls, or routing policy. `required_ai_provider` can require a provider while `execute_mission` marks a no-handler mission assigned; compatibility endpoints also return hard-coded 100% provider/tool reliability. Route generative work through the existing gateway, persist invocation evidence, and block provider-required assignments when no provider or executable tool exists.

## Two composition roots compete for agentic and analytics ownership

**Confirmed.** Root `packages/api/api/app.py::_mount_agentic_and_platform` mounts the all-in-one `api.agent_router` and a synthetic `/v1/analytics/executive-dashboard`. The modified UI is inside the nested `Website-Orchestrator` repository, whose `packages/api/api/app.py` mounts canonical planning, memory, runtime, reflection, agents, and SaaS analytics but not the CMO router. Its SaaS analytics router already owns `/v1/analytics/executive-dashboard` with a different contract. Selecting the root app disconnects the canonical stack; selecting the nested app leaves `/agentic/executive/*` absent; combining them creates route collisions.

The root mount catches every import or registration exception and silently omits agentic, CMO, copilot, and platform routes, turning startup defects into unexplained UI 404s. Establish one composition root and owner per route family, place CMO orchestration above the canonical runtime, and expose explicit degraded health when required routes cannot mount.

**Confirmed.** The local change spans a 77-file root diff, untracked CMO modules, and a nested repository whose changes appear to the root only as `Website-Orchestrator` gitlink state. Backend logs, helper scripts, and unrelated milestone work are bundled with it. Record the nested commit and split unrelated work before release.

## Existing tests do not exercise the CMO path

**Confirmed.** Root `uv run pytest` completed with `673 passed, 4 warnings in 298.51s`. No tests reference `AgentScheduler`, `CMOMemoryStore`, Executive Brain/report endpoints, connected-site CMO discovery, or mission execution. The untracked modules are imported only through an optional mount whose exception swallowing lets wiring defects evade smoke tests.

Not tested: cross-tenant run ownership; two websites under one tenant; per-site publishing credentials; policy precedence across governance modes and flags; concurrent `agent_config` updates; scheduler stop/start during a long cycle; multi-worker duplicate prevention; discovery failure preservation; report/UI schemas; simulated-versus-observed provenance; provider-required blocking; and accepted/applied/verified distinctions.
